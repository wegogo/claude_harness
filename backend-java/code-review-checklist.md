# Code Review 检查清单

> Java 后端 Code Review 参考。每次 PR 提交者和 Reviewer 都过一遍。
> 不是每项都要逐条勾选——按 PR 的变更范围选择性关注。

---

## 使用方式

- **提交者**：提交 PR 前自查，在 PR 描述中标注已检查项
- **Reviewer**：Review 时对照清单，发现问题留评论
- 关注变更相关项，不必每次都检查全部 60+ 项

---

## 一、基础检查（每个 PR 必过）

- [ ] 代码能编译通过，无 Error 级编译警告
- [ ] 所有测试通过：`./mvnw test`
- [ ] 新增/修改的功能有对应测试
- [ ] 没有提交调试代码（`System.out`、`printStackTrace`、注释掉的代码块）
- [ ] 没有硬编码的密码、密钥、Token、连接串
- [ ] PR 标题符合 Commit 规范（`type(scope): subject`）
- [ ] PR 关联了对应的 Issue/需求

---

## 二、功能与逻辑

- [ ] 代码实现了 PR 描述中的功能需求
- [ ] 边界条件已处理（null、空集合、0、负数、最大值）
- [ ] 异常路径已覆盖（数据库异常、网络超时、并发冲突）
- [ ] 幂等性：重复请求不会产生副作用（如涉及创建/支付）
- [ ] 并发安全：共享资源的读写是线程安全的
- [ ] 事务边界正确：`@Transactional` 加在正确的层级和范围

---

## 三、命名与可读性

- [ ] 变量名、方法名、类名见名知意，无歧义缩写（`usr` → `user`）
- [ ] 方法名描述行为而非实现（`getUsersByStatus` 而非 `getData`）
- [ ] 没有超过 80 行的方法，考虑抽取子方法
- [ ] 没有超过 4 层的嵌套，考虑提前 return 或抽取方法
- [ ] 魔法数字已提取为常量（`if (status == 3)` → `if (status == STATUS_ACTIVE)`）
- [ ] 复杂逻辑有注释解释"为什么"

---

## 四、架构与分层

- [ ] Controller 层没有业务逻辑（只有参数接收 → 调 Service → 返回）
- [ ] Controller 层没有直接操作数据库
- [ ] Service 层通过接口暴露能力，实现与接口分离
- [ ] 层间数据隔离：Entity 不直接返回给前端，用 DTO/VO 转换
- [ ] 没有循环依赖（A 依赖 B，B 又依赖 A）
- [ ] 公共工具方法抽取到 util 类，没有重复代码

---

## 五、异常处理

- [ ] 没有"吞掉异常"：`catch (Exception e) { }`（空 catch 块）
- [ ] 没有 `e.printStackTrace()`，使用日志记录：`log.error("xxx", e)`
- [ ] Controller 中没有 `try-catch`（交给全局异常处理器）
- [ ] 自定义异常用 `BusinessException`，携带明确的 `ErrorCode`
- [ ] 异常信息对用户友好，不暴露技术细节（SQL 异常堆栈等）

---

## 六、数据库与持久层

- [ ] 没有 `SELECT *`，显式指定查询字段
- [ ] 没有在循环中执行 SQL（N+1 查询问题）
- [ ] 大批量操作用了批量接口（`saveBatch` / `insertBatchSomeColumn` 而非循环 `insert`）
- [ ] 查询条件有索引支撑，`EXPLAIN` 验证过执行计划
- [ ] 没有拼接 SQL，使用了 MyBatis-Plus 条件构造器或 XML 参数化查询
- [ ] 表结构变更（DDL）有对应的 Flyway/Liquibase 脚本
- [ ] 分页查询有 `limit`，没有全表扫描

---

## 七、Redis 与缓存

- [ ] 所有缓存都设置了过期时间（`expire`）
- [ ] Key 命名遵循规范（`项目名:模块:业务:键`）
- [ ] 缓存更新策略正确（先更新 DB 还是先删缓存）
- [ ] 防缓存穿透：空值也缓存或用布隆过滤器
- [ ] 防缓存击穿：热点 Key 加锁
- [ ] 序列化使用 JSON，不是 JDK 序列化

---

## 八、日志

- [ ] 关键操作有日志（方法入口、重要分支、异常退出）
- [ ] 使用占位符 `{}`，没有字符串拼接
- [ ] 日志级别合理（INFO 业务关键点，WARN 可恢复异常，ERROR 系统级故障）
- [ ] 没有在循环中打高频日志（DEBUG 级除外）
- [ ] 敏感信息没有出现在日志中（密码、Token、身份证号）
- [ ] 异常日志记录了完整堆栈：`log.error("msg", e)`

---

## 九、安全

- [ ] 所有接口有鉴权（或显式在白名单中）
- [ ] 用户输入有校验和过滤（防 XSS、SQL 注入）
- [ ] 密码使用 BCrypt 加密存储，不是明文或 MD5
- [ ] 敏感数据传输走 HTTPS
- [ ] SQL 参数化查询，无拼接
- [ ] 文件上传有类型和大小限制
- [ ] 没有越权风险（用户 A 不能操作用户 B 的数据）

---

## 十、性能

- [ ] 没有在循环中做耗时操作（DB 查询、远程调用、IO）
- [ ] 大集合操作考虑了内存影响（流式处理或分批）
- [ ] 频繁读取的数据有缓存
- [ ] `@Transactional` 范围最小化，不在事务中做远程调用
- [ ] 避免大对象深拷贝，考虑不可变引用传递
- [ ] 定时任务有合理的并发控制和超时保护

---

## 十一、测试

- [ ] 新增/修改的代码有对应测试
- [ ] 测试覆盖了正常路径和异常路径
- [ ] 测试是独立的，不依赖执行顺序
- [ ] 测试中没有 `Thread.sleep()`（用 Awaitility）
- [ ] 测试命名清晰（`should_xxx_when_yyy` 或中文描述）
- [ ] 没有为了凑覆盖率写无意义的断言

---

## 十二、API 与接口

- [ ] 新增接口遵循 RESTful 规范（`/api/v1/`）
- [ ] 接口有 Swagger 注解（`@Operation`、`@Schema`）
- [ ] 请求和响应有明确的数据类型（DTO/VO）
- [ ] 破坏性变更（字段删除、类型改变）已通知调用方
- [ ] 分页接口返回统一分页结构
- [ ] 接口有合理的限流策略（如需）

---

## 十三、依赖与配置

- [ ] 新增依赖在 PR 中说明了用途
- [ ] 没有引入 SNAPSHOT 版本（生产代码）
- [ ] 依赖版本在 `dependencyManagement` 统一管理
- [ ] 配置项有合理默认值，不依赖环境变量硬编码
- [ ] 不同环境（dev/staging/prod）的配置分离

---

## Reviewer 评语模板

留评论时，按严重程度标注，方便提交者判断优先级：

| 标记 | 含义 | 示例 |
|------|------|------|
| `[Block]` | 必须修改，否则不合并 | 安全漏洞、逻辑错误 |
| `[Should]` | 强烈建议修改 | 性能问题、不规范写法 |
| `[Nit]` | 小建议，可选修改 | 命名优化、风格问题 |
| `[Question]` | 疑问，需讨论 | 设计选择、方案疑问 |
| `[Praise]` | 写得好的地方（鼓励正向反馈） | - |

### 评语示例

```
[Block] 这里 userId 没有做非空校验，如果前端传 null 会导致 NPE。
       建议在 DTO 加 @NotNull。

[Should] 这个方法在循环里查 DB，列表 100 条就是 100 次查询。
         建议改成批量查询 + Map 映射。

[Nit] 变量名 `u` 可以改成 `user`，更易读。

[Question] 这里用 ArrayList 还是 LinkedList？
           如果只是尾部追加，ArrayList 性能更好。

[Praise] 异常处理这块写得很清晰，每个分支都有对应的 ErrorCode。
```

---

## 快速 Review 流程

```
1. 先看 PR 描述，理解这次改了什么、为什么改
2. 跑一遍测试，确认能通过
3. 按文件 Review，先看接口/结构变更，再看实现细节
4. 对照清单的"基础检查"项
5. 按 [Block] → [Should] → [Nit] 顺序留评论
6. 确认通过后 Approve
```
